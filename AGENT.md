## What This Project Is

`privox` is a lightweight transparent proxy written in Rust. It sits between an AI agent
or application and any OpenAI-compatible LLM inference endpoint. It detects PII in outbound
prompts, replaces sensitive values with stable tokens, forwards the sanitized request
upstream, and restores original values in the response before returning to the caller.

The full requirements are in `docs/REQUIREMENTS.md`. Read that document before starting
any implementation work. If requirements and this file conflict, requirements win.
Note the conflict in SCRATCHPAD.md.

## Module Structure

Maintain this structure exactly. Do not create new top-level modules without noting the
decision in SCRATCHPAD.md.

```
src/
  main.rs           CLI parsing, config loading, server startup only. No business logic.
  config.rs         Config struct, TOML deserialization, env overrides, validation.
  server.rs         HTTP server, route handlers, request/response lifecycle.
  proxy.rs          Upstream HTTP client, streaming forwarder.
  detector/
    mod.rs          Detector trait, pipeline orchestration, entity type enum.
    regex.rs        All regex-based entity detectors.
    ner.rs          Optional NER client (Ollama). Opt-in only.
  tokenizer.rs      Token generation and determinism logic.
  detokenizer.rs    Response scanning, token substitution, streaming chunk handling.
  vault/
    mod.rs          Vault trait and public API.
    sqlite.rs       SQLite-backed vault implementation.
    crypto.rs       AES-256-GCM encrypt/decrypt, PBKDF2 key derivation.
  error.rs          All error types using thiserror.
  types.rs          Shared types: EntityType, TokenRecord, DetectedEntity, etc.

tests/
  integration/
    proxy_test.rs   End-to-end tests with mock upstream.
    streaming_test.rs
```

If a module approaches 400 lines, split it. Note the split in SCRATCHPAD.md.

## Code Rules

These are non-negotiable. No exceptions without a comment explaining why.

**Correctness**
- Zero compiler warnings. Zero clippy warnings. No suppressions without a comment.
- `cargo fmt` must produce no diff at all times.

**Error handling**
- Use `thiserror` for all error types in library code.
- Use `anyhow` for application-level error propagation in `main.rs` and `server.rs`.
- No `unwrap()` or `expect()` in non-test code unless the path is genuinely unreachable.
  If you use one, add a comment: `// SAFETY: <reason this cannot fail>`.
- Error messages must be actionable. "failed to open vault" is not actionable.
  "failed to open vault at {path}: {source}" is.

**Documentation**
- Every `pub` function, struct, enum, and trait must have a Rustdoc `///` comment.
- Every regex pattern in `detector/regex.rs` must have a comment citing the standard
  or rationale it implements and a human-readable description of what it matches.
- Complex or security-sensitive logic must have an inline comment explaining the
  reasoning, not just what the code does.
- `cargo doc --no-deps` must produce zero warnings.

**Security — hard rules, no exceptions**
- Original (pre-tokenization) values must NEVER appear in log output at any log level.
- Original values must NEVER appear in error messages returned to the caller.
- Original values must NEVER be stored anywhere except the encrypted vault.
- The vault must NEVER be exposed via any HTTP endpoint.
- If you are unsure whether something might log a sensitive value, it should not log it.

**Testing**
- Write the test before or alongside the implementation, never after.
- Every regex pattern needs at least one positive and one negative test case.
- Test data must use synthetic PII. Do not use real names, real emails, real card numbers.
  Use values like `test.user@example.com`, `4111111111111111` (standard test card), etc.
- Tests must use descriptive `assert_eq!(actual, expected, "description of what failed")`.
- No `todo!()` or `unimplemented!()` in code that is supposed to be complete.
  If something is intentionally deferred, use a `// TODO(next-session): <description>`
  comment and record it in SCRATCHPAD.md.

**Dependencies**
- Do not add a dependency that is not in the approved list in `docs/REQUIREMENTS.md`
  section 7.5 without recording the justification in SCRATCHPAD.md and adding a
  comment in `Cargo.toml`.
- Every entry in `Cargo.toml` must have a comment: `# <what this is used for>`.

## Configuration and Secrets

- The installation secret is stored at `~/.privox/secret.key` with permissions `0600`.
- If the secret file is missing at startup, the proxy refuses to start with a clear
  error. It does not silently generate a new one.
- Verify file permissions on startup. Refuse to start if they are weaker than `0600`.
- Never log the secret, the derived key, or any vault encryption material.

## Streaming

Streaming is a first-class feature, not an edge case.

- The detokenizer must handle tokens that span SSE chunk boundaries.
- Use a sliding window of at least 64 bytes when scanning streaming chunks.
- Forward chunks as they arrive. Do not buffer the entire stream.
- The `[DONE]` sentinel must be forwarded unchanged.
- Write integration tests for the streaming path. Do not assume the non-streaming tests
  are sufficient coverage.

## What Not To Build

Do not implement any of the following. They are explicitly out of scope for v1.
If the human asks for one of these, note it in SCRATCHPAD.md and confirm before starting.

- Multi-host vault replication or sync
- Authentication or API key management
- Model routing or load balancing
- Metrics export or tracing (structured logging only)
- Web UI or dashboard
- Plugin systems
- Per-user policy
- Rate limiting

Do not implement PresidioDetector before the Detector trait refactor is complete
and all existing tests pass. See REQUIREMENTS_ADDENDUM_001.md section 8.
